⇄ How a Request Flows

When someone visits eprom-portal.xyz/apps/heater/:

1
Browser sends request
Goes to your VM at 18.198.1.231
2
Nginx checks the URL path
/apps/heater/* → Heater (port 3001)
/apps/pump/* → Pump (port 3005)
/apps/massmole/* → MassMole (port 3006)
Everything else → Portal (port 3000)
3
App receives request + login cookie
The cookie is sent automatically by the browser
4
App checks: logged in? Has permission?
Reads cookie → checks database session → checks app permissions
5
Response sent back
Valid → show the app
Invalid → redirect to login

🔒 Authentication (How Login Works)

Here it is in Python pseudocode you'll understand:

# 1. User signs up password_hash = bcrypt.hash("their_password") # Stored in DB. Verification email sent. # 2. User logs in if bcrypt.verify(input_password, stored_hash): token = jwt.encode( {"user_id": 5, "role": "viewer"}, SECRET_KEY ) # Token goes into a COOKIE # (like a wristband at a concert) # Browser sends it automatically every time # 3. Every page checks the cookie token = request.cookies.get("auth_token") user = jwt.decode(token, SECRET_KEY) session = db.query( "SELECT * FROM sessions WHERE token_hash = $1", hash(token) ) if session and not expired: # Access granted!
Key point: One login on the Portal works for ALL apps (heater, pump, massmole). They all read the same cookie.

Security Measures

✓Passwords hashed with bcrypt (cost 12)
✓JWT tokens — signed, expire automatically
✓Cookies: httpOnly (JS can't steal them)
✓Rate limit: 5 login attempts per 15 min
✓Per-app permissions in database
✓Admin vs Viewer roles

🗃 Database Tables

TableStores
users Email, hashed password, role (admin/viewer), user number (USR-0001)
sessions Active logins — who's logged in, from where, when it expires
apps Registry of all apps (heater, pump, massmole, etc.)
user_app_permissions Which user can access which app (many-to-many)
activity_logs Audit trail — who did what, when

🐋 Docker Compose (The Recipe)

docker-compose.yml tells Docker how to run everything. In Python terms:

# What docker-compose.yml does (pseudocode): # Start the database first run_container("eprom_db", image="postgres:16", data_persists=True) # Then start each app (they all wait for DB) run_container("eprom_portal", build_from="./portal", port=3000) run_container("eprom_heater", build_from="./heater", port=3001) run_container("eprom_pump", build_from="./pump", port=3005) run_container("eprom_massmole", build_from="./massmole", port=3006) # All on a private network "eprom_net" # They talk to each other by name (e.g. "db:5432")
CommandWhat it does
docker compose up -d --build Start everything (rebuild if code changed)
docker compose down Stop everything
docker logs eprom_portal See portal logs (like print statements)

💬 Quick Answers for the Meeting

"What's the tech stack?"
Next.js portal + Express.js calculators + Python FastAPI pump + PostgreSQL, all in Docker on AWS EC2.
"How many apps?"
3 live (heater, pump, massmole), 3 planned (carbon, sensors, leak detection).
"How's security?"
JWT tokens in httpOnly cookies, bcrypt passwords, rate limiting, per-user per-app permissions, admin/viewer roles.
"Where's it hosted?"
AWS EC2 t3.micro in Frankfurt, behind Nginx reverse proxy. Domain: eprom-portal.xyz
"Does it use AI?"
Yes — each calculator app has a Claude AI chat assistant that can analyze results and give engineering recommendations.
"Single sign-on?"
Yes — one login on the portal gives access to all permitted apps via shared JWT cookie.
"Can it scale?"
Each app is a separate Docker container. Can move to bigger VM or split into multiple servers. Currently t3.micro (2 vCPU, 1GB RAM + 2GB swap).
"What about the database?"
PostgreSQL 16 with persistent storage. Tables for users, sessions, app permissions, activity logs. All queries use parameterized SQL (no injection risk).
