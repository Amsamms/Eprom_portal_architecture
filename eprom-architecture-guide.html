🔒 Authentication (How Login Works)

Here it is in Python pseudocode you'll understand:

# 1. User signs up password_hash = bcrypt.hash("their_password") # Stored in DB. Verification email sent. # 2. User logs in if bcrypt.verify(input_password, stored_hash): token = jwt.encode( {"user_id": 5, "role": "viewer"}, SECRET_KEY ) # Token goes into a COOKIE # (like a wristband at a concert) # Browser sends it automatically every time # 3. Every page checks the cookie token = request.cookies.get("auth_token") user = jwt.decode(token, SECRET_KEY) session = db.query( "SELECT * FROM sessions WHERE token_hash = $1", hash(token) ) if session and not expired: # Access granted!
Key point: One login on the Portal works for ALL apps (heater, pump, massmole). They all read the same cookie.

Security Measures

✓Passwords hashed with bcrypt (cost 12)
✓JWT tokens — signed, expire automatically
✓Cookies: httpOnly (JS can't steal them)
✓Rate limit: 5 login attempts per 15 min
✓Per-app permissions in database
✓Admin vs Viewer roles

1. What Is This Project?

Imagine a building with rooms. The building is a VM (virtual machine) running on AWS. Inside it, there are 6 rooms — each doing one specific job, sealed off from each other.

The Building = AWS EC2 VM (or Company VM)
The Receptionist = Nginx (decides who goes to which room)
The Filing Cabinet = PostgreSQL database (stores all data)
Room 1 = Portal (main website — login, dashboard, admin)
Room 2 = Heater (fired heater efficiency calculator)
Room 3 = Pump (centrifugal pump performance calculator)
Room 4 = MassMole (chemical mole/mass converter)
Room 5 = Optimizer (ML feature importance & parameter optimization)

Each room is a Docker container — a sealed box with its own tools and files. If one room crashes, the others keep running. They communicate over a private network called eprom_net, and the only way the outside world can reach them is through Nginx (the receptionist).

Browser → Nginx (port 80/443) | +→ / → Portal (port 3000) +→ /apps/heater/ → Heater (port 3001) +→ /apps/pump/ → Pump (port 3005) +→ /apps/massmole/ → MassMole (port 3006) +→ /apps/optimizer/→ Optimizer (port 3007) | All apps talk to → PostgreSQL (port 5432, internal only)

How Curve Fitting Works (Plain English)

Problem: You have 5-7 measured datapoints from the pump manufacturer (flow vs. head, flow vs. efficiency).
Solution: SciPy fits a polynomial curve through these points, so you can calculate the head at any flow rate, not just the measured ones.

Example: You measure H=100m at Q=0, H=95m at Q=50, H=80m at Q=100.
SciPy fits: H(Q) = 100 - 0.001*Q - 0.001*Q²
Now you can ask: "What's the head at Q=75?" → H(75) = 91.4m

5. MassMole App — Chemical Converter

Built with Express.js. Converts between mole% and mass% for gas mixtures using a database of 160 chemical compounds. Also calculates stream properties (average molecular weight, LHV).

The Mole/Mass Analogy

Imagine you have a bag of mixed coins: 50 pennies, 30 nickels, 20 dimes.

Mole% = "count basis" — 50% pennies, 30% nickels, 20% dimes (by number)
Mass% = "weight basis" — pennies are lighter than dimes, so the weight percentages are different

Molecular Weight is like the "weight of each coin type." Methane (CH4) is light (MW=16), while n-Butane (C4H10) is heavier (MW=58). Converting between mole% and mass% uses these molecular weights.

Why Hybrid Architecture?

Browser → Nginx → Node.js (port 3007) | +----------+----------+ | | /api/chat Everything else /api/health (proxied to PHP) (handled by Node) | v Apache/PHP (port 8080) Upload CSV, preprocess, run importance, optimize
Why not just use Node.js or just PHP?
The ML feature importance code already existed in PHP (from a friend's repo). Rather than rewriting it, Node.js acts as a "front door" that handles auth + AI chat (which all apps need), and proxies the data science to PHP. Best of both worlds.
What's the data pipeline?
Upload CSV → Preprocess (clean data) → Set Target Column → Run Feature Importance (Pearson correlation + elbow detection) → Run Optimization (random search, 10K iterations). Each step builds on the previous one.

9. How Security Works

The Login Flow (Step by Step)

1
User submits email + password
Rate limiter checks: has this IP tried more than 5 times in 15 minutes? If yes, block.
2
Server looks up user in database
Finds the bcrypt hash stored at signup. Uses bcrypt.compare() to check.
3
Creates a JWT token
Contains: {user_id: 5, role: "viewer", email: "..."}. Signed with SECRET_KEY.
4
Stores session in database
Saves SHA256(token) + expiry + IP address + user agent. This allows server-side logout.
5
Sets httpOnly cookie
Cookie flags: httpOnly (JS can't read it), secure (HTTPS only), sameSite: lax
6
Every subsequent request
Browser automatically sends the cookie. Server verifies JWT signature + checks session in DB.

Security Layers

LayerWhatWhy
bcrypt Passwords hashed (cost factor 12) Even if DB is stolen, passwords can't be reversed
JWT Signed tokens (can't be forged) Stateless auth — any app can verify without calling portal
httpOnly cookies JavaScript can't read the cookie Prevents XSS attacks from stealing tokens
DB sessions Token hash stored in sessions table Allows server-side logout (delete session = force logout)
Rate limiting 5 attempts per 15 min per IP Prevents brute-force password guessing
Per-app permissions user_app_permissions table User A can access heater but not pump
Subscription plans Basic/Professional/Enterprise Controls which apps + AI access each user gets
Parameterized SQL All queries use $1, $2 placeholders Impossible to inject SQL
Admin vs Viewer Role-based access control Only admins can manage users, view all logs, grant permissions
Key rule: JWT tokens are never returned in API response bodies — only set as httpOnly cookies. User role/permissions always come from the server (/api/auth/me), never trusted from localStorage.

10. How the AI Chat Works

Every calculator app has a Claude AI chat assistant. Here's how it works behind the scenes:

1
User types a message
Example: "What can I do to improve the thermal efficiency?"
2
Browser sends POST /api/chat
Includes: message, conversation ID, selected model, and the auth cookie
3
Server builds a system prompt
Includes: current calculation results, available tools, guardrails (safety rules)
4
Server calls Claude API
Sends conversation history + system prompt + tool definitions to Anthropic's API
5
Claude may use tools
Claude can call tools like calculate() to re-run formulas with different inputs, or get_field_reference() to look up formula details
6
Response streamed back via SSE
Words appear one-by-one in the chat UI (Server-Sent Events)
7
Everything logged to database
Message content, tool calls, model used, token counts, IP address

What Are "Tools"?

Tool Use is like giving Claude a set of functions it can call. Instead of just answering with text, Claude can say: "I need to run a calculation first", then call the calculate() tool with modified inputs, get the result, and then explain what changed.

Example flow:
User: "What if I increase excess air by 5%?"
Claude: calls calculate({...currentInputs, excess_air: currentValue + 5})
Claude: "Increasing excess air by 5% would reduce thermal efficiency from 89.2% to 87.8%..."

tool-executor.js ai/ Actually runs the tool when Claude asks (calls the engine)

11. How Deployment Works

There are two servers and multiple development machines. Code flows from development → staging (EC2) → production (company VM).

The Full Deployment Cycle

1
Edit code on laptop (or cloud)
Make changes in the local repo
2
rsync to EC2 (staging)

3
Build & restart on EC2
ssh ... "cd /opt/eprom && docker compose up -d --build portal"
4
Test on staging
Visit https://eprom-portal.xyz and verify everything works
5
rsync to company VM (production)
rsync -avz -e "sshpass -p '...' ssh ..." portal/src/ ai@192.168.50.202:/opt/eprom/portal/src/
6
Build & restart on production
Same docker compose command on company VM
7
git commit & push
Save to GitHub for backup and collaboration
NEVER use rsync --delete — it once destroyed critical VM-only files (db.js, auth.js, logout endpoint). Recovery was only possible because the Docker container was still running with the old code.
